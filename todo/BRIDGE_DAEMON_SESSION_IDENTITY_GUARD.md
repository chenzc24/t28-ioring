# Bridge Daemon Session Identity Guard

## Severity

Critical.

The bridge tunnel can be configured for one Unix user while the daemon behind the
forwarded remote TCP port belongs to a different Virtuoso session/user. This can
silently write generated schematics, layouts, symbols, and testbenches into the
wrong Cadence library path.

## Evidence From 2026-05-30

Local bridge profile `t28_io` was configured as:

```text
VB_REMOTE_HOST_t28_io=thu-wei
VB_REMOTE_USER_t28_io=chenzc_intern25
VB_JUMP_HOST_t28_io=thu-han
VB_JUMP_USER_t28_io=chenzc_intern25
VB_REMOTE_PORT_t28_io=65264
VB_LOCAL_PORT_t28_io=65271
```

The local SSH tunnel was:

```text
-L 65271:127.0.0.1:65264 chenzc_intern25@thu-wei
```

But the daemon responded from a different Virtuoso session:

```text
getShellEnvVar("USER")      => "zhangzixuan"
getShellEnvVar("HOME")      => "/home/zhangzixuan"
getShellEnvVar("PWD")       => "/home/zhangzixuan/TSMC28"
getWorkingDir()             => "/home/zhangzixuan/TSMC28"
ddGetObj("LLM_Layout_Design")~>readPath
  => "/home/zhangzixuan/TSMC28/tphn28hpcpgv18"
```

The configured project `cds.lib` was correct:

```text
/home/chenzc_intern25/TSMC28/llm_IO/cds.lib:
DEFINE LLM_Layout_Design /home/chenzc_intern25/TSMC28/llm_IO/LLM_Layout_Design
```

Therefore the tunnel configuration alone is not enough to prove the bridge is
attached to the intended Virtuoso process.

## Immediate Mitigation In This Repository

The T28 generator and simulator now fail before write operations if the active
Virtuoso library mapping disagrees with `_local/site.yaml` / configured `cds.lib`.
This prevents silent writes into another user's or another project's library.

## Required Bridge-Level Fix

`virtuoso-bridge-lite` should add a daemon identity handshake and enforce it in
`status`, `start`, and `VirtuosoClient.from_env()`:

1. The setup script loaded into CIW should record and expose:
   - Unix user: `getShellEnvVar("USER")`
   - home: `getShellEnvVar("HOME")`
   - working directory: `getWorkingDir()`
   - `CDS_LIB`: `getShellEnvVar("CDS_LIB")`
   - daemon profile name and expected remote port
   - a random per-start nonce generated by `virtuoso-bridge start`
2. The Python client should verify that the daemon response matches the selected
   profile's expected user/host/port/nonce before treating the daemon as healthy.
3. `status` should mark the daemon as failed if the tunnel user and daemon user
   differ, unless the user explicitly disables this check.
4. Remote daemon ports should be allocated per user/profile or checked for
   ownership before reuse.
5. The generated `load(".../virtuoso_setup.il")` instruction should include a
   visible profile/nonce banner so users can confirm they loaded the exact setup
   file into the intended CIW.

## Reproduction Shape

1. Another Virtuoso daemon is listening on the configured remote port.
2. Start a local tunnel with the expected user/profile.
3. `virtuoso-bridge status` reports `[tunnel]` as the expected user but `[daemon]`
   workdir/user from the other session.
4. Any code using only bridge reachability can operate on the wrong Cadence
   library table.

## Acceptance Criteria

- A tunnel to a daemon owned by the wrong Unix user fails `status`.
- `VirtuosoClient.from_env(profile="t28_io")` refuses to connect to a daemon whose
  identity does not match the selected profile.
- The status output distinguishes tunnel identity from daemon identity clearly.
- Existing workflows can still intentionally connect cross-user only with an
  explicit override.
